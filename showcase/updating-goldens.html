Visual goldens · operations runbook

Updating the golden screenshots, without the confusion

Two golden sets exist for different reasons, and three routes update them. Here is which one to reach for, when — and why “run it on CI and let it push” and “regenerate locally through Docker” are both valid answers to the same question.

Start here

The goldens come in two sets, not one. react/ is the x86 set CI enforces; react-local/<arch> is your machine’s native set for the instant local loop. An intentional UI change usually means updating both.

To refresh the CI set react/, pick a route — dispatch the GitHub workflow and let CI render & push it back (selective), or regenerate it locally in Docker and commit it yourself. Same bytes either way.

To refresh your local set react-local/<arch>, run the native :update script on your own machine. No Docker, instant — this is the loop the single-set experiment broke, and why we kept it.

Part 1 · the map

Two sets, two jobs

They render the same scenarios but live in different worlds. Font rasterisation drifts ~30% across CPU architectures, so an arm64 Mac cannot reproduce x86 pixels natively — hence a set for each.

react/ gates CI

The canonical x86 baseline. Rendered in the pinned Playwright container, checked by visual.yml on every push to main.

Part 2 · the routes

Three ways to update

Routes 1 and 2 both write the canonical react/ set — interchangeable, byte-for-byte. Route 3 writes your local set. Route 1 is the one that solved the 45-minute bottleneck.

Route 1

Selective CI refresh — and it pushes the commit for you

Dispatch update-visual-goldens.yml with a scenario_pattern. CI renders only the matching goldens in the pinned container, then auto-commits them straight back to your branch. One changed scenario costs about a minute, not thirty.

1

CI workflow — selective & self-committing

runs on: GitHub Actions · writes: react/

Trigger from the Actions tab (Run workflow) or the CLI. Empty pattern = full wipe & re-render; a pattern = targeted, no wipe. CI commits the result to your branch with [skip ci].

gh workflow run "Update visual goldens" \ --ref my-branch -f scenario_pattern=aurora
→ react/ selective ✓ CI commits & pushes ✓ no local Docker
2

Local Docker — regenerate & verify by hand

runs on: your machine (Docker) · writes: react/

The same pinned image under --platform linux/amd64 renders react/ byte-identical to CI, into your working tree — you review and commit. verify asserts the committed set: a CI-exact gate you can run before pushing.

pnpm goldens:regen # rewrite react/ into the tree pnpm goldens:verify # assert react/ passes (CI-exact)
→ react/ full set only (today) you commit needs Docker daemon
3

Local native — keep your fast loop green

runs on: your machine (no Docker) · writes: react-local/<arch>

Plain Playwright / Vitest on your host writes your native set. Instant, no container. Run it after an intentional UI change so pnpm test:ui:visual goes green again locally. Narrow it with the SCENARIO_PATTERN env var.

pnpm --filter @rtc/client-react \ test:ui:visual:playwright:react:update # tier: playwright (the sole CI-asserted tier)
→ react-local/<arch> instant ✓ env filter ✓ you commit
Part 3 · the decision

Which do I run?

Match the situation to the route. The common one — an intentional UI change — needs two sets updated, so it needs two routes.

I made an intentional UI change and both local and CI goldens are now stale.
Route 3 + (1 or 2)
Just my local pnpm test:ui:visual is red and I want it green again — nothing to push to CI yet.
Route 3
One or a few scenarios changed and I want CI to regenerate & commit the react/ set for me.
Route 1
I want the react/ set refreshed locally, no CI round-trip — or to reproduce a CI visual failure on my machine.
Route 2
The trap to avoid: updating only your react-local set (Route 3) and assuming CI is happy. CI checks react/, which Route 3 never touches. After an intentional visual change, refresh react/ too — via Route 1 or 2 — or visual.yml will redden on the next push to main.
