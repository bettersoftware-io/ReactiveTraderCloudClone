Cross-framework testing · how the Solid port is proven, not just built

One suite. React and Solid. One verdict.

@rtc/client-solid doesn't get its own test suite written twice — it gets proven against the same specs, the same goldens, and the same Gherkin features that already govern @rtc/client-react. Three mechanisms make that possible without either client importing the other.

Start here

Contract tier: the same *.contract.spec.ts files run twice, once per framework, through a driver seam (setDriver/getDriver) that swaps out only the rendering adapter.

Visual tier: Solid's Playwright and vitest-browser configs point their snapshot directory into client-react's own committed __screenshots__ tree and refuse to write — Solid owns zero goldens of its own, so a pixel match is a real cross-framework proof, not a tautology.

e2e tier: the same Gherkin features drive whichever client RTC_CLIENT_PKG names, spawned by the same devServer.ts — Solid's runs just point the dev-server spawn and the ports somewhere else.

The shape of it

One spec, two frameworks, one verdict

A spec (contract assertion, golden screenshot, or Gherkin scenario) never mentions React or Solid. It travels down two rendering paths and lands on the same pass/fail line either way.

Fine print

Where the proof has edges

The honest exceptions to "same suite, same verdict" — each one tracked, none of them silent.

e2e login.spec is Solid-excluded. Credential-wiring gap (client-solid doesn't read VITE_DEV_AUTH yet), not a missing LoginScreen — @rtc/client-solid shipped its own LoginScreen/AuthGate. See docs/STATUS.md.
Visual · CT 4 classic-skin CT assertions are CI-skipped. GitHub-hosted runners resolve classic's OS-generic font tokens non-deterministically across runs — a runner-environment issue, not a Solid rendering bug. Still verified by the other two x86 tiers and all darwin tiers.
Visual · CT tier Solid's "CT tier" is a full-page Playwright fallback. @playwright/experimental-ct-solid is frozen at 1.48.2 (its own component-test runner), so the fallback exercises the same scenarios through full-page Playwright instead.
Bindings useMachine disposes eagerly in Solid, deferred in React. React's version defers cleanup a microtask to survive StrictMode's dev-only double-invoke; Solid has no such double-invoke, so eager disposal is correct there — not a shortcut. See packages/solid-bindings/README.md.
