# Multi-producer GitHub Pages via a `gh-pages` branch

**Status:** approved design — 2026-07-14
**Author:** Claude (Opus 4.8) with @bettersoftware-io

## Problem

The repo needs to host two independent kinds of static artifact on GitHub Pages:

1. The **coverage report** — regenerated HTML from an on-demand build
   (`coverage-report.yml`), already live.
2. **Presentation decks** — self-contained standalone HTML (+ PDF) committed
   under `docs/presentations/<date>/`, LFS-tracked. Not on Pages today; GitHub
   cannot render a 4.3 MB LFS-pointer file, so real hosting is required.

GitHub Pages serves **exactly one site per repository**. The coverage workflow
currently owns it through the *artifact* model (`upload-pages-artifact` +
`deploy-pages`), where each deploy replaces the entire site. A second,
independent producer using the same model would clobber the first. We want the
two to publish **independently**, each owning a subtree, with neither able to
wipe the other.

## Goals

- Two (and any future N) producers publish to one Pages site without collision.
- Coverage keeps working throughout the migration — **zero downtime**.
- Presentations publish **automatically** when a deck lands on `main`.
- No new external GitHub Action; no Vercel; minimal new dependency surface.
- LFS-stored deck bytes are published (not the pointer file).

## Non-goals

- No Vercel / external hosting.
- No change to coverage's *generation* logic — only its publish step.
- Coverage stays **manual** (`workflow_dispatch`); we do not auto-run it.
- No per-deck manifest / authoring convention beyond "name the file nicely".

## Architecture

Switch Pages from the artifact model to **"Deploy from a branch → `gh-pages` /
(root)"**. The branch is a pure deployment artifact (not a source branch). Each
producer owns one subtree and only ever writes within it, so independent commits
never lose each other's content.

```
gh-pages (branch, served at https://bettersoftware-io.github.io/ReactiveTraderCloudClone/)
├── .nojekyll                     # raw static serving; no Jekyll processing
├── index.html                    # root hub — SOURCE lives in main: docs/pages/index.html
├── coverage/…                    # owner: Coverage Report workflow
└── presentations/
    ├── index.html                # GENERATED by the publish workflow (deck list)
    └── 2026-07-14/
        ├── Clean-Architecture-case-study.html
        └── Clean-Architecture-case-study.pdf
```

### Ownership map (the invariant that prevents collisions)

| Path                     | Owner                        | How written                    |
|--------------------------|------------------------------|--------------------------------|
| `/.nojekyll`, `/index.html` | `docs/pages/**` in `main`  | synced by the publish workflow |
| `/coverage/**`           | `Coverage Report` workflow   | rsync `_site` → `coverage/`     |
| `/presentations/**`      | `Publish Site` workflow      | sync decks + generate index    |

### URL changes after cutover

- Root `/` → the **hub** (new).
- Coverage moves root → **`/coverage/`** (approved). The old root URL now shows
  the hub, which links straight to `/coverage/`.
- Decks at **`/presentations/<date>/<file>.html`** (+ `.pdf`).

## Components

### 1. Root hub (source of truth in `main`)

`docs/pages/index.html` — a small static landing page linking to *Coverage
Report* (`./coverage/`) and *Presentations* (`./presentations/`). `docs/pages/`
also carries `.nojekyll`. Kept in `main` so it is reviewable and versioned; the
publish workflow syncs it to the branch root. Rarely changes.

### 2. `Publish Site` workflow (new) — owns root + `/presentations/`

- **Trigger:** `push` to `main` filtered to
  `paths: [docs/presentations/**, docs/pages/**]`, plus a `workflow_dispatch`
  fallback (republish without a new commit).
- **Permissions:** `contents: write` only.
- **Concurrency:** `group: pages-write`, `cancel-in-progress: false` (see Races).
- **Steps:**
  1. `actions/checkout` **with `lfs: true`** — materialises the real deck bytes,
     not the 3-line LFS pointer.
  2. Check out `gh-pages` into a worktree; **create it as an orphan if missing**
     (self-bootstrapping — no separate bootstrap job needed). The create-if-
     missing check is idempotent (test for the remote branch first) and runs
     under the `pages-write` concurrency guard, so two first-runs cannot both
     create the branch.
  3. Sync `docs/pages/*` → branch root; sync `docs/presentations/*` →
     `presentations/`. This is a **full, idempotent sync of the entire
     `docs/presentations/` tree** — every committed deck is (re)published on
     each run, regardless of which path triggered the workflow. So a push that
     touches only `docs/pages/**` still republishes all decks, and re-runs are
     safe.
  4. **Generate `presentations/index.html`**: enumerate `presentations/<date>/`
     folders; for each deck, derive the display title from the **filename**
     (`Clean-Architecture-case-study` → "Clean Architecture case study"), take
     the date from the folder name, and link the `.html` and any sibling `.pdf`.
  5. Commit + push via `GITHUB_TOKEN`, with a `git pull --rebase` retry loop.

### 3. `Coverage Report` workflow (migrated) — owns `/coverage/`

- **Unchanged:** the whole coverage-generation section and `_site` assembly.
- **Replaced tail:** drop `configure-pages`, `upload-pages-artifact`, and the
  separate OIDC `deploy` job. Instead: check out `gh-pages` (orphan-if-missing),
  rsync the assembled `_site` → `coverage/`, commit + push via `GITHUB_TOKEN`
  (same rebase-retry + `pages-write` concurrency group).
- **Permissions shrink:** `pages: write` + `id-token: write` → `contents: write`.
- **Trigger unchanged:** stays `workflow_dispatch`.
- The assembled `_site/index.html` becomes `/coverage/index.html` — the coverage
  section landing, linked from the root hub.

## Cross-cutting concerns

### Races (two writers, one branch)

- Both workflows share `concurrency: { group: pages-write, cancel-in-progress:
  false }`. Concurrency groups are **repo-global by resolved name**, so the two
  different workflows queue against each other rather than racing.
- Belt-and-suspenders: each push is wrapped in a `git pull --rebase` + retry
  loop, so a non-fast-forward (from a concurrent writer) is resolved and
  retried rather than failing.
- Because each writer only touches its own subtree, even an interleaved commit
  never drops the other producer's files.

### LFS

Only the publish workflow needs `lfs: true` (it copies committed deck bytes).
Coverage regenerates its own content and needs no LFS. Verification must confirm
the deck on the branch is ~4.3 MB of real HTML, not a 3-line pointer.

### CI-trigger safety (no loops, no waste)

- `ci.yml` triggers on `push: branches: [main]` only — `gh-pages` pushes do not
  match, so publishing never kicks off CI.
- `GITHUB_TOKEN`-authored pushes do not trigger workflows at all (GitHub's
  loop-prevention) — a second layer of safety.

### Permissions / the one manual step

Flipping the Pages **source** (artifact model → branch model) is a Pages-admin
operation. This repo's fine-grained PAT is expected to `403` on the Pages-admin
API (consistent with its other admin 403s), so this is a **one-time ~30-second
UI action by the maintainer**: Settings → Pages → Source → *Deploy from a
branch* → `gh-pages` / `(root)`.

## Migration — zero-downtime cutover

The live coverage site keeps serving the **old artifact** until the final flip,
so nothing breaks mid-migration:

1. Land the new workflows + `docs/pages/` scaffold on `main` (via PR/CI/merge).
2. Run `Publish Site` (dispatch or the merge itself) → creates the `gh-pages`
   orphan, populates root hub + `/presentations/`.
3. Run `Coverage Report` (new branch mode) → populates `/coverage/`.
4. **Inspect** the `gh-pages` branch: deck HTML is real bytes (~4.3 MB, not a
   pointer), `presentations/index.html` links resolve, `/coverage/` present,
   `.nojekyll` at root.
5. **Maintainer flips the Pages source** to `gh-pages` / (root).
6. **Verify live:** curl the root hub, `/coverage/`, and the deck URL — expect
   `200` + expected content.

## Testing / verification strategy

- **Pre-flip (all we can prove without Pages serving from the branch):**
  inspect the `gh-pages` tree; validate generated `presentations/index.html`
  (titles derived correctly, links present); assert the deck blob size is the
  real artifact, not a pointer.
- **Post-flip:** curl the three live URLs for `200` and content assertions.
- Workflow YAML sanity: the repo pins all actions by SHA — new `actions/checkout`
  usage must carry a pinned digest + `lfs: true`.
