# Plan A — Client Build/Test Toolchain Upgrade Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Upgrade the `@rtc/client` build/test toolchain — `vite` 6→8, `@vitejs/plugin-react` 4→6, `vitest` 3→4 (across all 5 packages), `jsdom` 25→29 — in one coordinated bump, landing on a fully green workspace.

**Architecture:** `vitest` is a devDependency in all five workspace packages (`domain`, `shared`, `server`, `client`, `tests`), so it moves to `^4` everywhere at once; `vite`/`@vitejs/plugin-react`/`jsdom` are `@rtc/client`-only. The known vitest-4 breakage (the `WsAdapter` WebSocket mock) is fixed *first* in a forward-compatible way that still passes on vitest 3, de-risking the bump. After bumping, each verification surface (typecheck + unit tests, build, visual, e2e) is driven to green as its own task. Remaining fallout from vite 8 / plugin-react 6 / jsdom 29 is discovered per-gate and fixed via systematic debugging.

**Tech Stack:** pnpm workspaces + Turborepo, Vite 8, Vitest 4, @vitejs/plugin-react 6, jsdom 29, React 19, Playwright 1.60 (visual + e2e), Node v24.16.

Facts that govern this plan:

- **`@rtc/client` is the only package with `vite`, `@vitejs/plugin-react`, `jsdom`.** Those three bump in `packages/client/package.json` only.
- **Peer facts (verified 2026-06-06):** `vitest@4.1.8` peers `vite ^6||^7||^8`; `@vitejs/plugin-react@6` peers `vite ^8`; `vitest@4` peers `jsdom: *`. Node v24.16 satisfies `vitest@4` (`>=24`) and `vite@8` (`>=22.12`) engines.
- **Client test config** is `packages/client/vitest.config.ts` (`environment: "jsdom"`, `include: src/**/*.test.ts(x)`). **Client build config** is `packages/client/vite.config.ts` (`plugins:[react()]`, dev server on `PORT||5173`, `build.outDir:"dist"`). The visual tiers have their own Vite configs under `visual/` (host + CT `ctViteConfig`).
- **The whole effort runs on a feature branch off `main`** (the dependency-upgrade work; do NOT start on `main`). If a post-`pnpm install` error like `Cannot find module @rollup/rollup-linux-…`/`@rollup/rollup-darwin-…` appears, re-run `pnpm install` — known pnpm optional-dep quirk, not a code defect.

## Task 2: Make the WsAdapter WebSocket mock vitest-4 compatible (on vitest 3 first)

This is the one breakage already diagnosed: vitest 4 constructs a stubbed global with `new`, and the current stub's `mockImplementation(() => …)` arrow is not a constructor. Fixing it as a constructable class works on **both** vitest 3 and 4, so we land it now, green, before the bump.

**Files:**
- Modify: `packages/client/src/app/adapters/WsAdapter.test.ts`

- [ ] **Step 1: Replace the arrow-function stub with a constructable class stub**

In `packages/client/src/app/adapters/WsAdapter.test.ts`, make `MockWebSocket` capture the latest instance in its constructor, and stub the global with the class itself (a real constructor). Replace lines 6–34 (the `class MockWebSocket { … }`, the `let lastMock`, and the `beforeEach`) with:

```ts
let lastMock: MockWebSocket;

class MockWebSocket {
  static OPEN = 1;
  static constructed = 0;
  readyState = 0;
  onopen: ((ev: Event) => void) | null = null;
  onclose: ((ev: CloseEvent) => void) | null = null;
  onmessage: ((ev: MessageEvent) => void) | null = null;
  onerror: ((ev: Event) => void) | null = null;
  send = vi.fn();
  close = vi.fn();
  constructor() {
    MockWebSocket.constructed++;
    // Capture the most recently constructed socket so tests can drive its
    // event handlers. WsAdapter does `new WebSocket(url)`, so a constructable
    // stub is required — vitest 4 forwards `new` to the stubbed global, and an
    // arrow `mockImplementation` is not a constructor.
    lastMock = this;
  }
}

beforeEach(() => {
  vi.useFakeTimers();
  MockWebSocket.constructed = 0;
  // The class carries a static OPEN (=1) like the real WebSocket constructor,
  // so WsAdapter.send()'s readyState check behaves as it does in a browser/Node.
  vi.stubGlobal("WebSocket", MockWebSocket);
});
```

Leave the `afterEach` (lines 35–38) and all `it(...)` blocks unchanged — they reference only `lastMock` and `MockWebSocket.constructed`, both preserved. The adapter passes `url` to `new WebSocket(url)`; `MockWebSocket`'s constructor ignores the argument, which is fine.

- [ ] **Step 2: Run the WsAdapter suite on the current vitest 3 to confirm it still passes**

Run:
```bash
pnpm --filter @rtc/client exec vitest run src/app/adapters/WsAdapter.test.ts
```
Expected: **all tests pass** (the file has 10 `it`s across the describe block). The constructable stub is behaviour-equivalent on vitest 3.

- [ ] **Step 3: Run the full client suite to confirm no regression**

Run:
```bash
pnpm --filter @rtc/client test
```
Expected: green (26 files / 69 tests at baseline).

- [ ] **Step 4: Commit**

