- **WebSocket connects before login — FIXED.** DevTools Network showed the deployed app repeatedly attempting (and failing) the `wss://` connection *before* the user had logged in. **Root cause (confirmed, not the originally-guessed subscribe timing):** `WsAdapter`'s *constructor* called `connect()` eagerly, and `buildBrowserPorts` constructs the adapter during `createApp()` — i.e. in `AppRoot`'s first render pass, before `AuthGate` renders the login screen. Pre-login `sessionStore.read()` is `null`, so `buildWsUrl` emitted the bare URL with no `?access=` param; the server's `verifyClient` → `authorizeUpgrade` is strict (no open-when-empty fallback) and rejected it 401; `onclose` then rescheduled the same tokenless upgrade every 3s for as long as the user sat on the login screen. No Rx subscription was involved — gating a *subscribe* would not have fixed it. Secondary finding: nothing reopened the socket on a successful login either, so sign-in only worked because the next 3s retry happened to pick up the freshly-minted token — a **up-to-3s dead window** after login. **Fix:** `WsAdapter` gained an `autoConnect` option (default `true`) plus `connect()`/`disconnect()`; all three clients construct with `autoConnect: false` and pass the adapter as the new optional `AppPorts.transport`; `createApp` drives it from `AuthPresenter.state$` (connect on `authenticated`, disconnect otherwise), which also connects synchronously for a resumed session and closes the socket on sign-out. Covered by `WsAdapter.test.ts` ("auth gating") and `composition.authGate.test.ts`.
  - **Residual (web verified, RN not):** the pre-login-then-connect behaviour was verified live against the real stack on **web only** (zero WS attempts while logged out, then exactly one carrying `?access=`, with live prices rendering; reverting `autoConnect` reproduces the tokenless pre-login upgrade). `client-react-native`'s `buildNativePorts.ts` got the identical `autoConnect: false` + `transport` wiring and its jest suite passes, but the actual on-device/simulator behaviour is **unverified** — the gate lives in shared `client-core` so it should hold, but nobody has watched a logged-out RN app to confirm no socket opens pre-login.   - **Knock-on:** this likely also resolves the *pre-auth* half of the blocked `gatewayUnauthorized → auto-logout` item above — a never-authenticated client no longer retries at all, so an opaque reconnect loop now unambiguously means "authenticated token went stale / server unreachable" rather than "never had a token." The mid-session-expiry case is unchanged and still blocked on a protocol signal.

- **Blank line between adjacent multi-line "block" statements — PARKED (investigated, no rule adopted).** Spun off from the `rtc/newspaper-order` extension (PR #287) that now pushes `jest.mock`/`jest.unmock` below the tests like `vi.mock` (both are hoisted above imports by their runners, so position is behaviour-neutral). The newspaper autofix already *separates* the mocks it moves (it joins them with a blank line), but there is **no rule that enforces a blank line between adjacent multi-line blocks in general** — e.g. two `jest.mock(…, () => {…})` factories authored already-at-the-bottom, packed with no separator. Goal was a *generic, size-based* rule ("if a statement spans more than ~2–3 lines, separate it"), **not** an identifier-based mock-only one. Findings: **(1) Biome cannot do it** — the maintainers deliberately don't implement `padding-line-between-statements`; blank-line *insertion* is classed as formatting, and the Biome formatter only preserves/collapses blank lines, never inserts required ones ([biomejs/biome#3016](https://github.com/biomejs/biome/discussions/3016)). **(2) ESLint can, via the existing `padding-line-between-statements`** (already in `eslint.config.mjs`): the `multiline-expression` selector matches exactly the missing case — multi-line call statements ending in `)` (like `jest.mock(…)` / `describe(…)`) that the current `multiline-block-like` clause misses because they don't end in `}`. Config would be one clause: `{ blankLine: "always", prev: "multiline-expression", next: "multiline-expression" }`. **(3) But its blast radius is large and imprecise** — measured on the repo it flags **350 spots across 152 files** (120 test, 32 production), all autofixable, with two false-positive modes: **(a)** simple statements that merely *wrapped* because they were long (e.g. a 3-line `expect(headers.get("CT")).toBe("application/json")`) get split from the next assertion — same spirit as the "don't separate one-liners" objection; **(b)** production RxJS assignment chains (`this.x$ = src.pipe(…)`) get separated field-by-field. Root cause: no built-in selector expresses *"a call whose **argument** is a real block / object-literal body"* — `multiline-block-like` keys on a trailing `}`, `multiline-expression` on any 2+-line span; the granularity is binary (single- vs multi-line), with no line-count threshold. **Options if revived:** (i) accept the generic `multiline-expression` clause (repo-wide, or scoped to `**/*.{spec,test}.{ts,tsx}` to spare production RxJS) and live with the wrapped-statement noise; (ii) write a small autofixable custom `rtc/` rule that fires only between multi-line statements whose body is an actual block/object-literal (exempting merely-wrapped ones) — precise, matches the "block" intent, but new code to maintain; (iii) do nothing and rely on the newspaper-order autofix for the common (moved-mock) case. **Parked** until it recurs / annoys enough to justify option (i) or (ii).
