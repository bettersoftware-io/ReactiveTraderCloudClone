## ⚪ Optional / next step (no plan file yet)

- **Auth UI on Solid** — the Solid client has no `LoginScreen`/`AuthGate` (react-only; the two `shell/auth` contract specs, the `LoginScreen.module.css` parity-allowlist entry, and the e2e `login.spec.ts` exclusion in `tests/browser/playwright/playwright.config.ts` all key off this). Porting it would complete auth parity and drop all three exclusions. Not spec'd.
- **`/devtools/` serving on Solid** — the Solid client wires the devtools instrumentation (hub/decorators, shipped with the port) but its Vite config never serves the inspector SPA at `/devtools/` the way client-react does (dev-only asset edge to `@rtc/devtools-app`); the e2e `devtools.spec.ts` is excluded for solid runs on that basis. Wiring it is a small vite-config + build-order change. Not spec'd.
- **Classic-skin fonts are host-environment-sensitive in CT visual crops** — classic is the only theme skin whose font tokens resolve to OS-generic keywords (`system-ui`/`ui-monospace`) rather than embedded `@fontsource` faces, so element-cropped screenshots of text-width-driven surfaces vary with the runner's system fonts — and GitHub-hosted runners resolve them NON-DETERMINISTICALLY across runs (observed 2026-07-17: react's own CT tier rendered classic chrome/header 1218px in three runs and 1177px in another, flaking against its own goldens; likely since the ubuntu24 runner-image rollout). Surfaced by the SolidJS-port Phase 4 assert-only tiers (PR #230), which skip 4 classic CT assertions ON CI ONLY in solid's ct-fallback tier (still verified by the other two x86 tiers and all darwin tiers). Real fix: back classic's font tokens with a deterministic embedded face in the VISUAL HARNESS (app appearance unchanged for users) — or app-wide as a design change — then regen classic goldens across tiers/platforms. Until then react's post-merge visual.yml can flake on classic CT crops.
- **Auth Phase 2 — per-user server state isolation** — not yet spec'd. Phase 1 (login + genuine server-side auth) shipped PR #210; the RN login screen + [authentication.md](authentication.md) shipped PR #226; the three deferred follow-ups (client-tracked server `exp`, RN AsyncStorage session persistence, RN sign-out control) shipped 2026-07-17. Phase 2 would add a per-connection `ctx` keyed by the authenticated user — per-user blotter/orders/positions with shared hot market-data streams — replacing today's process-wide shared simulator singletons. Phase 1's signed token + `AuthPresenter` are the seam.
  - **Blocked (not actionable as-is):** `gatewayUnauthorized → auto-logout` — when a session token expires mid-session the WS reconnect fails its `?access=` upgrade check, but a browser cannot observe the 401 on a WebSocket *upgrade* rejection (the handshake just fails opaquely), so the client can't distinguish "server asleep" from "token rejected" and auto-return to the login screen. No clean fix without a protocol change (e.g. a server close-code or a pre-connect token-validity ping). Today an expired session surfaces as a stuck reconnect until the user manually signs out; the stored-session `exp` check (PR #234) bounds how long a stale token is even offered.
- **`pnpm dev` → `dev:web` with feature-flag client selection** — `pnpm dev` is currently a plain alias of `dev:react` (React web client, simulator mode). The intended end state is to rename it to `dev:web` and have it pick the web client implementation (React vs Solid) from a feature flag, while `dev:react` / `dev:solid` remain the explicit per-implementation entry points. Documented in [CLAUDE.md](../CLAUDE.md) (Build Commands → "Future direction"); not spec'd or built. The SolidJS-port workstream has now reached full client parity (contract + visual + e2e), so this item is unblocked — it awaits only its own feature-flags spec (see the Feature flags item above).
